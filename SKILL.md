## 一、核心能力

本技能掌握以下专业能力：

### 1.1 商品搜索（选品）
- **自然语言理解**：用户的自然语言描述直接作为 query 传给 API，API 自行理解
- **丰富数据分析**：API 返回每个商品的 stats 数据，包括销量趋势、复购率、好评率、量价关系等，Agent 可据此给出选品建议
- **结果格式化**：将 API 返回转换为含关键指标的商品列表，同时输出原始 stats 数据供深度分析

### 1.2 店铺管理
- **绑定店铺查询**：获取用户已绑定的下游店铺列表
- **授权状态诊断**：识别店铺授权是否过期，给出处理建议
- **平台特性了解**：熟悉各平台的入驻要求和运营特点

### 1.3 铺货执行
- **批量铺货**：支持单次最多50个商品的批量铺货
- **渠道适配**：自动匹配商品到对应平台格式
- **结果追踪**：区分成功/失败，提供重试建议

### 1.4 用户引导
- **新手引导**：AK获取 → 开店绑定 → 首次选品 → 首次铺货
- **问题诊断**：根据错误信息判断原因并给出解决方案
- **最佳实践**：基于场景提供选品和铺货建议

## 二、知识体系

### 2.1 认证机制（AK）

AK（Access Key）是访问 1688 平台的身份凭证。

#### 获取 AK

1. 下载 1688 AI版 APP
2. 登录后进入「我的」→「开发者设置」
3. 复制 AK（一串字母数字组合，通常40+位）

#### 配置 AK

本 skill 通过 **OpenClaw SecretRef** 读取 AK。配置一次，所有会话生效。

**方式一：对话中配置（推荐）**

直接在飞书/对话中告诉我：
```
我的AK是 xxx...xxx
```

我会立即：
1. ✅ 通过 `configure.py` 保存 AK 到配置文件（持久化）
2. ✅ 在当前会话的后续命令中手动注入 AK（`ALI_1688_AK=xxx python3 ...`）
3. 🔄 **继续你的选品/铺货请求**

当前会话即可看到结果。**重启 Gateway 后**，所有新会话自动注入，无需再手动传递。

**方式二：手动配置 openclaw.json**

编辑 `~/.openclaw/openclaw.json`：

```json5
{
  // ... 其他配置
  skills: {
    entries: {
      "1688-skills": {
        env: {
          ALI_1688_AK: "your_ak_here"
        }
      }
    }
  }
}
```

然后重启 Gateway。

**方式三：使用配置脚本**

```bash
cd {baseDir}/scripts
python3 configure.py your_ak_here
```

**方式四：Gateway 启动参数**

```bash
# ~/.zshrc 或 ~/.bashrc
export ALI_1688_AK=your_ak_here

# 使配置生效
source ~/.zshrc
```

⚠️ **注意**：方式四（export）只对从当前 shell 启动的 Gateway 有效。如果 Gateway 作为系统服务运行（LaunchAgent/systemd），请使用方式一、二或三。

#### 安全提示

- AK 仅用于本地 API 调用，不会上传到任何服务器
- 建议定期更换 AK（可在 1688 AI版 APP 中重新生成）
- 不要将 AK 提交到代码仓库

### 2.2 下游渠道特性

| 渠道 | 特点 | 入驻门槛 | 适合商品 | 注意事项 |
|------|------|---------|---------|---------|
| **抖音** | 内容电商，流量大 | 需营业执照 | 新奇特、视觉冲击力强的商品 | 重视视频素材 |
| **拼多多** | 低价走量 | 个人可入驻 | 性价比高的日用百货 | 价格敏感 |
| **小红书** | 种草社区 | 需品牌资质 | 美妆、生活方式 | 重视图文质量 |
| **淘宝** | 传统电商 | 需保证金 | 全品类 | 竞争激烈 |

### 2.3 选品策略

#### 好卖商品的特征（结合 stats 数据判断）
1. **应季性**：当前季节需要的商品（如夏天卖风扇）
2. **销量验证**：`last30DaysSales` 高且 `saleQuantityList` 趋势向上
3. **供应链稳定**：`collectionRate24h` 高（发货快）、`totalOrder` 大
4. **质量可靠**：`goodRates` 高、`repurchaseRate` 高、`remarkCnt` 多
5. **竞争空间**：`downstreamOffer` 不宜过高（同行铺货少 = 蓝海）
6. **利润空间**：通过 `tradePriceList` 分析成交价区间，评估定价空间

#### 避坑指南
- ❌ 避免侵权商品（品牌仿品）
- ❌ 避免过重/过大商品（运费成本高）
- ❌ 避免保质期短的食品
- ❌ 避免 `goodRates` 低或 `remarkCnt` 极少的商品（售后风险高）
- ❌ 避免 `saleQuantityList` 持续下降的商品（正在退潮）

### 2.4 常见错误及处理

| 错误现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| "AK未配置" | 环境变量未设置 | 提示用户配置 ALI_1688_AK |
| "店铺授权过期" | 长时间未使用或密码修改 | 引导用户在1688 AI版APP重新授权 |
| "选品返回为空" | 关键词太冷门或无货源 | 建议更换关键词或扩大搜索范围 |
| "铺货失败" | 商品信息不完整或平台审核 | 检查商品详情，等待审核或换品 |
| "网络超时" | API响应慢或网络问题 | 自动重试，最多3次 |

## 四、API 使用参考

### 4.1 核心脚本

所有脚本位于 `{baseDir}/scripts/` 目录，通过 bash 在该目录下执行：

**选品**：
```bash
cd {baseDir}/scripts && python3 search.py --query "帮我在1688找支持一件代发包邮的连衣裙，100元以内的" --channel douyin
```

**查店铺**：
```bash
cd {baseDir}/scripts && python3 shops.py
```

**铺货**：
```bash
cd {baseDir}/scripts && python3 publish.py --shop-code "260391138" --item-ids "123,456"
```

**检查 AK 配置状态**：
```bash
cd {baseDir}/scripts && python3 configure.py
```

**配置 AK**：
```bash
cd {baseDir}/scripts && python3 configure.py your_ak_here
```

### 4.2 核心数据结构

```python
# Product属性: id, title, price, image, url, stats(dict|None)
# Shop属性: code, name, channel, is_authorized
# PublishResult属性: success, published_count, failed_items
```

**stats 字段详解**（选品 API 返回，部分商品可能无 stats）：

| 字段 | 含义 | 选品价值 |
|------|------|---------|
| `totalSales` | 累计销量 | 判断商品热度 |
| `last30DaysSales` | 近30天销量 | 判断近期趋势 |
| `last30DaysDropShippingSales` | 近30天下单量 | 一件代发活跃度 |
| `repurchaseRate` | 复购率 | 高复购 = 产品质量好 |
| `goodRates` | 好评率 | 售后风险评估 |
| `remarkCnt` | 评价数量 | 样本量参考 |
| `collectionRate24h` | 24h揽收率 | 发货速度指标 |
| `downstreamOffer` | 下游铺货数 | 竞争激烈度 |
| `totalOrder` | 累计下单笔数 | 历史交易规模 |
| `categoryName` | 商品类目 | 类目归属 |
| `categoryListName` | 三级类目 | 精确类目 |
| `earliestListingTime` | 最早上架时间 | 新品/老品判断 |
| `saleRangeList` | 面价范围趋势 | `[{date, minPrice, maxPrice}]` |
| `saleQuantityList` | 销量趋势 | `[{date, saleQuantity}]` |
| `tradePriceList` | 交易量价关系 | `[{date, tradeCount, minPrice, minPriceQuantity, maxPrice, maxPriceQuantity}]` |

### 4.3 结果展示规范（必须遵守）

> **核心规则**：脚本输出的 `markdown` 字段是预渲染的展示内容，Agent **必须原样展示**，不得重新排版、删减或替换。Agent 的分析建议必须**追加在 markdown 之后**，不得混入其中。

#### 选品结果展示模板

脚本输出 JSON 含 `products`、`data_id`、`markdown`。Agent 必须按以下结构展示：

```
第一部分（必须）：原样输出 markdown 字段内容
─────────────────────────────────────
第二部分（必须）：Agent 选品分析（格式见下方模板）
─────────────────────────────────────
第三部分（必须）：引导下一步
```

**选品分析模板**（Agent 基于 `products[].stats` 填充，结构固定，内容因数据而异）：

```
📊 选品分析

🌟 推荐铺货（N个）：
1. [商品标题] — ¥[价格]
   30天销量 [X] | 好评率 [X%] | 复购率 [X%] | 铺货数 [X]
   → [一句话推荐理由，必须引用具体数据]

2. ...

⚠️ 风险提示：
- [商品标题]：[具体风险，必须引用 stats 数据]
- ...

需要铺货哪些商品？还是想调整搜索条件？
```

**分析维度**（Agent 从以下维度选择与用户意图最相关的组合）：

| 维度 | stats 字段 | 判断标准 |
|------|-----------|---------|
| 爆款潜力 | `last30DaysSales` + `saleQuantityList` | 销量高且上升趋势 |
| 供应稳定 | `collectionRate24h` + `totalOrder` | 揽收率高、成交量大 |
| 质量保障 | `goodRates` + `remarkCnt` + `repurchaseRate` | 好评率/复购率双高 |
| 竞争程度 | `downstreamOffer` | 低=蓝海，高=红海 |
| 价格趋势 | `saleRangeList` / `tradePriceList` | 价格波动和利润空间 |

#### 店铺结果展示模板

脚本输出 JSON 含 `total`、`valid_count`、`expired_count`、`shops`、`markdown`。Agent 按以下结构展示：

```
第一部分（必须）：原样输出 markdown 字段内容
─────────────────────────────────────
第二部分（按需）：若 expired_count > 0，提醒用户重新授权
第三部分（必须）：引导用户选择目标店铺
```

#### 铺货结果展示模板

脚本输出 JSON 含 `success`、`markdown`。Agent 按以下结构展示：

```
第一部分（必须）：原样输出 markdown 字段内容
─────────────────────────────────────
第二部分（按需）：
  - success=true → "还需要铺货其他商品吗？"
  - success=false → 分析失败原因 + 给出重试建议
```

#### 禁止行为

- ❌ 不得忽略 `markdown` 字段，用原始 JSON 数据自行排版
- ❌ 不得删减 `markdown` 中的商品条目或指标
- ❌ 不得将分析内容插入 `markdown` 商品列表中间
- ❌ 不得在没有 stats 数据支撑的情况下给出选品建议

### 4.4 完整链路示例（选品 → 查店 → 铺货）

## 五、最佳实践

### 5.1 对话节奏控制

**不要一次性给太多信息**：
- ❌ "这是10个商品，你可以铺货到抖音店或拼多多店，也可以继续搜索其他商品..."
- ✅ "找到10个商品，先看下这些是否符合需求？"

**每步确认后再继续**：
1. 选品完成 → 问"满意吗？"
2. 确认满意 → 问"铺货到哪个店铺？"
3. 确认店铺 → 执行铺货 → 展示结果

### 5.2 错误恢复策略

**网络错误**：
- 自动重试3次（已实现）
- 仍失败 → 提示"网络不稳定，请稍后再试"

**认证错误**：
- AK无效 → 引导重新配置
- 店铺过期 → 引导重新授权

**业务错误**：
- 选品无结果 → 建议换关键词
- 铺货失败 → 分析具体原因

### 5.3 用户体验优化

**首次使用**：
- 耐心解释每个概念（AK、店铺、铺货）
- 提供详细的步骤指引
- 允许用户随时提问

**熟练用户**：
- 快速执行，减少确认步骤
- 记住偏好（常用渠道、店铺）
- 提供快捷指令

## 六、边界与限制

### 6.1 本技能不做的事

- ❌ 不替代用户在电商平台的后台操作
- ❌ 不保证商品一定能卖出
- ❌ 不处理售后和客服问题
- ❌ 不提供宏观市场趋势预测（但可基于商品 stats 分析单品趋势）

### 6.2 需要用户自己完成的

- 在电商平台注册账号
- 处理买家订单和售后
- 决定定价策略
- 遵守平台规则和政策

### 6.3 技术限制

- 选品返回数量由 API 决定，markdown 摘要最多展示20个，完整数据在 JSON 的 products 字段
- 单次铺货最多50个商品
- API调用频率受限于1688平台
- 部分功能需要有效的AK和授权
